## Bugs found and fixed in this PR

### B1. Two-pass-verified GCP hits silently dropped — `stravart/crossref.py`

**Symptom.** Route #1272 (St Albans Shark): OCR cleanly read 4 streets,
cross-ref correctly clustered them in St Albans/UK with 95% confidence,
**but `_gcps_from_ocr` produced only 2 GCPs** — failing the affine-fit
minimum of 3.

**Root cause.** The two-pass-verify recovers the cluster via
`ways_named_in(name, city, country)` for streets where the worldwide
top-40 didn't include the right city — but stores those hits only in
the `verified_streets` local. The returned `CrossRefResult.matches`
still contains only the worldwide top-40. Downstream,
`reconstruct._gcps_from_ocr` builds GCPs by intersecting `matches` with
the cluster bbox; for common-name streets ("Victoria Street") none of
the 40 worldwide hits land in the small St Albans bbox, so the anchor
silently disappears.

**Fix.** Splice the per-candidate verified hits back into `matches`
(additive — worldwide hits preserved) before returning.

**Validation.** New regression test
`test_two_pass_recovery_splices_hits_back_into_matches` asserts the
splice for the exact St Albans-style scenario. Suite: 167 pass (166 + 1).

### B2. OSMnx Overpass calls fail with SSL on macOS — `stravart/mapmatch.py`

**Symptom.** Every route that reached map-match in run #1 failed with
`SSLError: certificate verify failed: self-signed certificate in
certificate chain` from `overpass-api.de`. 4/4 routes that got past
georef died here.

**Root cause.** `crossref.py` already handles this for httpx by feeding
its own SSL context; OSMnx routes through `requests`/`urllib3`, which
ignore the httpx context. On corporate-proxied macOS hosts the system
root store carries a self-signed cert that `certifi`'s bundle does not.

**Fix.** At `mapmatch` import time, set `REQUESTS_CA_BUNDLE` and
`SSL_CERT_FILE` env vars to the same keychain dump
`crossref._macos_keychain_bundle()` already produces and caches.
`requests` honours those env vars on every connection.

**Validation.** Loaded a real Cambridge walking graph (470 nodes, 1206
edges) from live Overpass through the fix. (See run #2 ship rate
below.)

## Dataset-level failure class: regional / marathon-zoom views

**Pattern.** Routes #5 (Manchester Dog), #36 (West Devon 100k tour) —
both are regional-scale runs. The strav.art rendering at that zoom
shows **place / town labels** (Stockport, Warrington, Cheadle, etc.)
rather than street labels. EasyOCR returns 50+ fragments per image but
none of them parse as a street name (none have a `Road / Street /
Avenue / Lane / …` suffix), so `OcrResult.street_candidates` is empty
and the pipeline aborts at "ocr: no street candidates".

**Why it can't be fixed in this PR.** The contour extraction works
fine; what's missing is a *geographic anchor*. For these routes the
title parser already gave us a city hint (`Manchester`, `Devon`),
which is what Phase 1 used. A natural Phase 4b would be: **when OCR
finds zero street candidates but title-derived `lat/lon` is populated,
fall back to a centroid-anchored projection** (no per-pixel affine —
just a fixed-scale, fixed-center placement using the cluster's known
city as the center). The contour shape is preserved; geographic
fidelity is lower but not zero.

**Scope estimate.** From the 1,654-row catalog, regional/marathon-zoom
views look to be ~5-15% based on title text. Cheap to gate later via
`OCR.n_street_candidates == 0 AND (title-derived city is known)`.

## Recommended tuning knobs for the 1,654-route scale-up

These are ordered by expected ship-rate impact.

1. **Don't lower `min_streets` below 3 yet.** The 3-anchor minimum is what
   keeps the affine fit non-degenerate. Several routes had 5-15 OCR
   candidates and the bottleneck was *whether each candidate's hit
   landed in the cluster bbox* — fixed by B1 above, not by lowering
   `min_streets`.

2. **Pad `cluster_radius_km` to 5.0 for cities >1M pop.** Default 3.0
   creates a tight bbox that excludes legitimate "Victoria Street"-class
   names that sit on the city edge. Risk: lets in nearby suburb hits.
   Mitigation: the RANSAC affine-fit drops outliers by residual.

3. **Keep `bbox_pad_m` at the default 200 m** for OSM graph downloads;
   when the projected polyline falls near the city edge, the snap might
   miss a few peripheral streets. Bumping to 500 m has minimal latency
   cost (Overpass is bottlenecked on tile size, not bbox by ~2×).

4. **Hold `min_confidence` at 0.6 by default**, but lower to 0.4 for a
   "low-confidence inspect" tier. The geometric mean punishes weak
   stages quickly: a 5-GCP / 80-m-RMSE / 0.4-fidelity fit lands at
   ≈0.45, which the 0.4 threshold accepts as "review-worthy".

5. **Set `waypoint_step_m` to 50 m for marathon-scale runs.** At 30 m
   the Dijkstra-per-segment count climbs faster than the snap accuracy
   improves; 50 m halves wall-clock with <5% fidelity loss.

6. **Add a fallback path for `OcrResult.street_candidates == [] AND
   title-derived city != None`** (the regional/marathon class). Out of
   scope for Phase 4a but probably the single biggest dataset-level
   ship-rate lever for the catalog at large.

## Wall-clock numbers (this run)

- Cold cache: ~25-30 s OCR + ~5-15 s OSMnx graph + ~1-3 s map-match
  + ~1.1 s × N Nominatim queries (rate-limited) per image.
- Warm cache: drops to ~25-30 s (OCR-bound). EasyOCR doesn't cache
  intermediate features per-image.
- For 1,654 routes from cold start, expect ~12-15 hours wall-clock as
  the Phase 3 handoff predicted. The Nominatim cache built up by
  Phase 4a is reusable.

## What's next

* **Phase 4b (regional/marathon class).** Implement the centroid-anchored
  fallback projection for routes where OCR yields zero street candidates
  but the title parser already gave a city. Probably +10-15% to the
  scaled ship rate.
* **Phase 4c (ranking).** Add per-confidence-tier review queues so the
  iOS client can serve only ≥0.7 by default and let users opt into
  ≥0.4. Pipes the four `_confidence` components into the row so the
  client can show *which* dimension is weak.
* **Phase 4d (geocoding feedback).** Routes that ship a high-fidelity
  GPX should write `geocode_source = 'reconstruct'` back to the row,
  enabling the iOS client's R-tree search to surface them.
